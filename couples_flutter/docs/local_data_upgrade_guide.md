# 本地数据继承与安全升级说明

## 当前结论

当前项目使用 Drift + SQLite，本地数据库名称固定为 `couples_local_db`，位于应用私有目录中。

在以下条件同时满足时，本地数据可以继承：

1. Android 包名不变
2. 使用同一签名证书
3. 通过“覆盖安装 / 正常升级安装”更新应用，而不是先卸载再安装

满足以上条件时，`release -> release` 升级安装会继承本地 Drift 数据，聊天记录、记账、待办、歌单、课表等本地数据不会因为升级被主动清空。

## 容易导致数据清空的情况

### 1. 卸载后重装
卸载应用会删除应用私有目录，本地 Drift 数据会一起被系统清空。

### 2. debug -> release
在 Android 上，`debug` 包和 `release` 包通常不是同一签名。
这类情况下多数设备不能直接覆盖安装，常见做法是先卸载 debug 包再安装 release 包，这一步会导致本地数据丢失。

### 3. 修改包名
如果包名变化，系统会把它当成另一个应用处理，原有本地数据库不会被继承。

## 当前数据库迁移状态

当前 `schemaVersion = 12`，迁移链已经覆盖旧版本到当前版本：

- `< 2` 创建 `poke_events`
- `< 3` 创建 `todos`、`todo_progress`
- `< 4` 为 `song_reviews` 增加评分字段

- `< 7` 创建 `feed_events`
- `< 8` 创建本地身份/情侣表，并为聊天补充 `senderUserId`、`clientMessageId`
- `< 9` 为聊天增加 `messageType`、`mediaUrl`、`mediaDurationMs`

当前迁移逻辑是“逐版本补表/补字段”，没有发现会主动删表、重建全库或清空数据的代码路径。

## 为什么升级后聊天可能出问题，但不等于本地数据被清空

本次排查里，聊天页报错的直接原因不是 Drift 数据丢失，而是客户端把“输入状态 typing 接口失败”当成了聊天页加载失败。

也就是说：

- 文字聊天的云端列表/发送主接口仍然可用
- 本地聊天表也没有被主动清空
- 但因为额外调用了后端未实现的 typing 接口，导致页面整体报错

这类问题会表现成“聊天页空白 / Failed to load chat data”，看起来像数据没了，实际根因是接口兼容性而不是数据库继承失败。

## 推荐的安全升级方式

### Android 本地测试
- 同一个 debug 包连续安装：可以继承本地数据
- 同一个 release 签名包连续覆盖安装：可以继承本地数据

### 准备给真实设备长期使用时
建议尽快固定：

1. 包名
2. release 签名
3. 升级安装方式

实际操作建议：

- 不要通过“卸载旧包再装新包”来更新
- 尽量使用同签名 APK / AAB 覆盖安装
- 每次发新版本前先确认 `schemaVersion` 与迁移逻辑完整

## 当前最小保护策略

当前版本先采用最小可运行保护策略，不引入复杂恢复架构：

1. 保持数据库名稳定：`couples_local_db`
2. 保持 Drift 迁移链连续，不跳版本删除旧结构
3. 保持本地身份、情侣关系、聊天、待办、歌单、课表等都走本地持久化
4. 对后端新增但未落地的能力（如 typing）做客户端兼容降级，避免“接口缺失导致页面整体不可用”

## 后续建议

如果后面要继续增强“升级后更稳”的能力，建议优先级如下：

1. 导出一份本地 JSON 备份
2. 增加启动时的数据库健康检查
3. 对关键表做轻量恢复提示

当前这一步先不引入这些额外架构，优先保证现有本地数据在正常升级安装下可以继承。
